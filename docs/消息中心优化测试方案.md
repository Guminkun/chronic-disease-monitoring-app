# 消息中心优化测试方案

## 📋 需求分析

### 功能需求
1. **消息展示优化**
   - ✅ 用户可以看到所有成员的消息（已实现）
   - 🔧 优化跳转逻辑：点击消息后切换成员，跳转到对应模块
   
2. **消息处理流程**
   - 🆕 添加"已处理"按钮
   - 🆕 已处理的消息不展示，流转到历史消息
   - 🆕 添加"历史消息"标签页

3. **消息状态管理**
   - 未读 → 已读（当前已有）
   - 未处理 → 已处理（新增）

## 🗄️ 数据库设计

### 修改 notifications 表

```sql
-- 添加字段
ALTER TABLE notifications 
ADD COLUMN is_handled BOOLEAN DEFAULT FALSE,
ADD COLUMN handled_at TIMESTAMP WITH TIME ZONE,
ADD COLUMN handler_type VARCHAR(50); -- 'user' | 'system'

-- 索引优化
CREATE INDEX idx_notifications_handled ON notifications(is_handled);
CREATE INDEX idx_notifications_patient_handled ON notifications(patient_id, is_handled);
```

### 数据模型更新

```python
# backend/app/models.py
class Notification(Base):
    __tablename__ = "notifications"
    
    # ... 现有字段 ...
    
    is_handled = Column(Boolean, default=False)  # 是否已处理
    handled_at = Column(DateTime(timezone=True))  # 处理时间

// frontend-mobile/src/api/notification.ts
export function markNotificationAsHandled(notificationId: number) {
  return request({
    url: `/notifications/${notificationId}/handle`,
    method: 'PUT'
  })
}

export function markAllNotificationsAsHandled(memberId?: string) {
  return request({
    url: '/notifications/handle-all',
    method: 'PUT',
    params: memberId ? { member_id: memberId } : {}
  })
}
```

#### 步骤3：实现成员切换逻辑
```typescript
// 跳转前切换成员
const handleView = async (item: NotificationItem) => {
  // 1. 切换成员
  if (item.member_id) {
    await memberStore.switchMember(item.member_id)
  }
  
  // 2. 标记已读
  if (!item.is_read) {
    await markNotificationAsRead(item.id)
  }
  
  // 3. 跳转到对应页面
  if (item.type === 'medication') {
    uni.switchTab({ url: '/pages/medication/index' })
  } else if (item.type === 'revisit') {
    uni.navigateTo({ url: '/pages/revisit/plan' })
  }
}
```

## 📌 注意事项

1. **数据迁移**：确保现有消息的 is_handled 默认为 false
2. **权限验证**：验证用户只能处理自己及家庭成员的消息
3. **成员切换**：跳转前确保成员切换成功
4. **消息去重**：避免重复生成相同消息
5. **性能优化**：大量消息时考虑分页和缓存
